## 性能瓶颈总结

| 瓶颈 | 影响 | 优先级 |
|------|------|--------|
| GPU线程同步阻塞（ondemand搬运） | GPU过载，CPU空闲29.6% | P0 |
| CPU时间表前6项估算不准确 | 决策可能错误 | P1 |
| 热点策略占用GPU过多 | 并行度低（1.62x vs 1.77x理论） | P2 |
| Placeholder数量限制（6个） | Ondemand并发受限 | P3 |

## 关键发现

1. **Microbench测量**: e/tg准确，但cpu_time_table前6项是估算值（不可靠）
2. **预取效果**: 占比仅3.8%，对吞吐影响微弱（热点专家已在GPU）
3. **负载均衡**: GPU过载（负载比1.30:1），CPU空闲29.6%
4. **主要瓶颈**: ondemand搬运在GPU线程内同步执行，增加GPU负担

1. 预取策略当前对性能有轻微负面影响，吞吐率下降约2%-7%

## 关键发现

1. **门控预测本身准确 (84.7%)**，但预取命中率仅 0.59-1.45%，说明预测准确的专家大部分已在 GPU 热表
2. **预取对命中率几乎无贡献**：bs=1 时预取命中 0.59%，bs=4 时 1.45%
3. **并行度随 batch_size 提升**：bs=1 时 1.06x（CPU 工作太少），bs=4 时 1.47x（负载更均衡）
4. **bs=4 时 GPU/CPU 各约 20ms**，并行时间约 24ms，理论最优 2x，实际 1.47x，开销约 26%
5. **重构后吞吐对比**：bs=1 时 1.56→1.65 t/s（预取ON），bs=4 时 4.01→3.89 t/s（预取ON 反而略降）

## 预取效果差的根因分析

| 原因 | 占比 | 说明 |
|------|------|------|
| 热表已覆盖 73% | 主要 | 门控预测准确的专家大部分已在 GPU 热表，不需要预取 |
| 调度模式限制 | 次要 | Decode Mode C（两层都 ≥ n_g）时不预取，而 674/1664 常驻使大多数层进入 C |
| 预取配额限制 | 次要 | max_ondemand=4，且受带宽预算约束 |

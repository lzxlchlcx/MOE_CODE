**总体概览**
本次重构对应最近提交 `f355b93 refactor: 解耦 DeepSeek MoE 运行时`。核心目标是把原来集中在 `src/model/deepseek.py` 里的 DeepSeek MoE 运行逻辑拆成多个职责明确的模块：

- `deepseek.py`：保留模型加载、推理主流程、Transformer 层级前向传播编排。
- `expert_types.py`：定义专家调度/执行过程中共享的数据结构。
- `placeholder_manager.py`：管理 GPU placeholder、专家驻留状态、加载中状态和淘汰。
- `expert_scheduling.py`：负责把当前层专家需求和预测的未来专家需求转换成调度计划。
- `expert_executor.py`：根据调度计划实际并行执行 CPU 专家、GPU 专家和预加载。
- `expert_predictor.py`：预测下一层可能会用到的专家。
- `expert_latency.py`：封装 CPU/GPU/拷贝延迟估计。
- `tests/test_runtime_refactor.py`：覆盖调度结果不重叠、placeholder 被视作 GPU 驻留等关键行为。

当前工作区是干净的，没有未提交变更。

**文件职责**
`src/model/deepseek.py`

这是运行时入口，类是 `FiddlerDeepSeek`。它现在主要负责：

- 加载 HuggingFace DeepSeek 模型和 tokenizer。
- 初始化 GPU、dtype、cache、beam search 参数。
- 初始化 `ExpertPlaceholderManager`。
- 初始化调度策略：`GPUOnlyStrategy` 或 `PrefetchHybridStrategy`。
- 初始化 `GatePredictor`、`ExpertLatencyModel`、`ExpertExecutionManager`。
- 把非专家模块加载到 GPU。
- 根据显存估算静态驻留专家数量。
- 执行 `generate()` 和 `mixtral_forward()` 主推理流程。

重构后，`deepseek.py` 不再直接承担大部分专家调度和执行细节，而是只负责“编排”。

`src/model/expert_types.py`

这是本次重构的类型中心。它定义了多个模块之间传递的结构化对象：

- `ExpertKey`：专家唯一标识，包含 `layer` 和 `expert_id`。
- `ExpertDemand`：某个专家的需求，包含 token 数、得分、来源。
- `ExpertAssignment`：某个专家对应的 token index 和 routing weight。
- `PlacementSnapshot`：当前专家驻留状态快照，包括静态 GPU、placeholder、loading、CPU、SSD、剩余 placeholder 数。
- `ExpertLayerRequest`：调度器输入，包含当前层、阶段、当前需求、未来需求、token 分配。
- `ExpertSchedule`：调度器输出，包含 CPU 执行、GPU 执行、preload、evict。
- `ExpertLayerContext`：执行器输入，包含当前层专家模块、flatten 后的 hidden states、hidden dim、专家分配。

同时提供辅助函数：

- `build_current_demands()`：从当前活跃专家生成当前层需求。
- `build_assignments()`：把原始 `(token_indices, routing_weights)` 转成 `ExpertAssignment`。
- `build_future_demands()`：从预测的专家张量生成未来专家需求。
- `unique_demands()`：去重未来需求。

这个文件的意义是让调度、预测、执行、placeholder 管理之间不再依赖临时 tuple 和隐式约定。

`src/model/placeholder_manager.py`

负责 GPU placeholder 生命周期管理。它现在维护几类状态：

- `_available`：空闲 placeholder id。
- `_occupied`：placeholder id 到 `(layer, expert_id)` 的映射。
- `_reverse_map`：`(layer, expert_id)` 到 placeholder id 的反向映射。
- `_static_gpu_resident`：初始化时常驻 GPU 的专家。
- `_loading`：正在加载中的专家，避免重复 preload。
- `_cpu_resident`：当前在 CPU 上的专家。
- `_ssd_resident`：预留给 SSD 层级的驻留集合，目前更多是结构占位。

关键方法：

- `acquire_placeholder()`：获取 placeholder，必要时触发淘汰。
- `acquire_free_placeholder()`：只使用空闲 placeholder，不触发淘汰，主要用于机会型 preload。
- `load_weights()`：把 CPU expert 的权重拷贝到 placeholder。
- `mark_static_gpu_resident()`：标记专家静态常驻 GPU。
- `mark_cpu_resident()`：标记专家位于 CPU。
- `mark_loading()` / `unmark_loading()`：管理加载中状态。
- `is_on_gpu()`：判断专家是否在静态 GPU 或 placeholder 中。
- `snapshot()`：生成 `PlacementSnapshot`，供调度器只读决策。
- `release_by_layer()`：释放某一层的 placeholder。

这里引入了锁 `_lock`，说明 placeholder 状态会被并行执行路径访问，例如专家执行和 preload 同时运行。

`src/model/expert_scheduling.py`

这是调度策略核心。文件内包含两层设计：

第一层是旧接口风格的 `ExpertSchedulingStrategy`：

- `GPUOnlyStrategy`：所有活跃专家都走 GPU。
- `HybridCPUGPUStrategy`：通过 CPU/GPU 成本穷举决定专家执行位置。
- `PrefetchHybridStrategy`：当前主要使用的 PDScope 风格预取混合调度策略。

第二层是更干净的新调度抽象：

- `ExpertScheduler`：只定义 `schedule(request, placement, latency)`，输入是结构化请求，输出是 `ExpertSchedule`。
- `PDScopeScheduler`：真正实现 prefill/decode 感知调度。

关键辅助函数：

- `_build_expert_mask()`：把 `selected_experts` 转成 one-hot mask。
- `_collect_active_experts()`：找出有 token 分配的专家。
- `_organize_token_assignments()`：整理每个专家对应的 token 和 routing weight。
- `_build_latency_lookup()`：从 benchmark 数据生成延迟表。
- `_lookup_latency()`：按 token_count 查询延迟。

`PDScopeScheduler` 的核心逻辑：

- `schedule()`：根据 `request.phase` 分发到 `schedule_prefill()` 或 `schedule_decode()`。
- `schedule_prefill()`：结合当前非驻留专家和未来预测专家，选择哪些当前专家上 GPU、哪些留 CPU、哪些未来专家 preload。
- `schedule_decode()`：根据 decode 阶段的当前驻留数量和下一层驻留数量，进入 A/B/C 或 fallback prefill 模式。
- `_select_global_queue()`：构造全局候选队列。
- `_select_current_ondemand()`：从当前专家里选出值得按需加载到 GPU 的专家。
- `_select_preload()`：根据 CPU/GPU 时间差和 placeholder 容量选择可预加载的未来专家。

`PrefetchHybridStrategy` 是目前 `deepseek.py` 调用的适配层。它负责：

- 从 gate 输出构造当前专家需求。
- 接收预测出的未来专家需求。
- 接收 `placeholder_manager.snapshot()`。
- 构造 `ExpertLayerRequest`。
- 调用 `PDScopeScheduler.schedule()`。
- 把 `ExpertSchedule` 转回 `deepseek.py` 仍然使用的返回格式：`cpu_expert_ids, gpu_expert_ids, preload_expert_ids, raw_assignments`。

`src/model/expert_executor.py`

这是实际执行专家计算的模块。核心类是 `ExpertExecutionManager`。

它的入口方法是：

```python
execute(schedule: ExpertSchedule, context: ExpertLayerContext) -> torch.Tensor
```

执行时会用 `ThreadPoolExecutor(max_workers=3)` 并行启动最多三类任务：

- `execute_gpu_experts()`：执行调度到 GPU 的专家。
- `execute_cpu_experts()`：执行调度到 CPU 的专家。
- `preload_experts()`：预加载下一层预测专家到 placeholder。

执行完成后：

- GPU 结果留在 GPU。
- CPU 结果拷回 GPU。
- 两者相加，返回当前 MoE 专家输出。

`execute_gpu_experts()` 的执行路径：

- 如果专家已经静态驻留 GPU，直接调用 `context.experts[expert_id]`。
- 如果专家已经在 placeholder 中，直接调用 placeholder。
- 如果不在 GPU，则申请 placeholder、加载权重、执行、再释放当前层 placeholder。
- 最后按 token index 做 `index_add_` 聚合。

`execute_cpu_experts()` 的执行路径：

- 把当前专家对应 token 的 hidden states 移到 CPU。
- 调用 CPU 上的 expert。
- 乘 routing weight。
- 在 CPU result 上做 `index_add_`。
- 最终由 `execute()` 拷回 GPU。

`preload_experts()` 的执行路径：

- 按 layer 聚合 preload demands。
- 跳过已经在 GPU/placeholder 中的专家。
- 跳过正在 loading 的专家。
- 使用 `acquire_free_placeholder()`，也就是不为了 preload 淘汰已有 placeholder。
- 加载成功后更新 preload 统计。

`src/model/expert_predictor.py`

这里定义专家预测接口：

- `ExpertPredictor`：抽象基类。
- `GatePredictor`：当前实现。

`GatePredictor.predict()` 的逻辑很直接：

1. 根据当前层 `current_layer` 和 `lookahead` 找到下一层。
2. 调用下一层的 `mlp.gate(hidden_states)`。
3. 得到 `predicted_experts` 和 `routing_weights`。
4. 调用 `build_future_demands()` 转成 `List[ExpertDemand]`。

也就是说，目前预测器不是训练出的单独模型，而是复用下一层 gate 对当前 hidden states 做前瞻估计。

`src/model/expert_latency.py`

封装延迟模型：

- `cpu(token_count)`：查询 CPU 执行延迟。
- `gpu_compute(token_count)`：查询 GPU 计算延迟。
- `transfer(layer, expert_id)`：返回专家权重传输延迟，目前统一为 `t_io`。
- `gpu_total(token_count, resident, layer, expert_id)`：GPU 计算加可能的传输成本。

调度器不直接关心 benchmark JSON 的具体结构，而是通过这个模型查询成本。

`tests/test_runtime_refactor.py`

目前有两个轻量测试：

- `test_scheduler_returns_disjoint_current_experts()`：确保调度器不会把同一个当前专家同时放进 CPU 和 GPU，并且当前专家集合完整。
- `test_placeholder_snapshot_treats_placeholder_as_gpu()`：确保被 placeholder 占用的专家在 snapshot 和 manager 里都被视作 GPU 驻留。

```python
config = transformers.AutoConfig.from_pretrained(args.model)
```

```python
template_expert = first_layer_mlp.experts[0]
```

```python
self.expert_loc = np.zeros(...)
```

```python
self.set_expert_loc(n_expert_on_gpu)
```

```python
self.bring_expert_to_gpu()
```

```python
self.expert_executor = ExpertExecutionManager(...)
```

**推理总流程**
入口是：

```python
FiddlerDeepSeek.generate()
```

整体顺序：

1. 重置 KV cache：

```python
self.past_key_value = transformers.cache_utils.DynamicCache()
self.past_key_values_length = 0
```

2. 处理输入文本，调用 `tokenize()`：

```python
input_ids, position_ids, attention_mask = self.tokenize(text, input_token)
```

3. 进入生成循环：

```python
for i_token in range(output_token):
```

4. 每轮根据阶段区分 prefill 和 decode：

- 第一轮是 prefill。
- 后续轮是 decode。
- decode 阶段会重建 attention mask，使其长度等于 `past_key_values_length + 1`。

5. 构造新的 `position_ids` 和 `cache_position`。

6. 调用核心前向：

```python
logits = self.mixtral_forward(
    input_ids,
    new_position_ids,
    attention_mask,
    cache_position,
    is_prefill=not is_decode,
)
```

7. 对 logits 做 softmax 和 top-k/beam 选择。

8. 更新下一轮输入 token。

9. 累计 prefill/decode 时间。

10. 输出最终生成结果和专家命中率。

**单次 `mixtral_forward()` 调用顺序**
`mixtral_forward()` 是本次重构后最关键的编排函数。

执行顺序如下：

1. 输入 token 进入 embedding：

```python
inps = self.model.embed_tokens(input_ids)
```

2. 计算 rotary position embedding：

```python
position_embeddings = self.rotary_emb(inps, position_ids)
```

3. 使用 transformers 官方工具构造 causal mask：

```python
causal_mask = create_causal_mask(...)
```

4. 遍历每一层 transformer：

```python
for i_layer, layer in enumerate(self.model.layers):
```

5. 第 0 层特殊处理：

- 第 0 层是完整 attention + dense MLP，不走 MoE expert 调度。
- 执行顺序是 input layernorm、self attention、残差、post attention layernorm、dense MLP、残差。
- 然后 `continue` 进入下一层。

6. 从第 1 层开始进入 MoE 层流程：

- input layernorm。
- self attention。
- 残差连接。
- post attention layernorm。
- gate 计算当前 token 的 top expert：

```python
selected_experts, routing_weights = layer.mlp.gate(inps)
```

7. 启动下一层专家预测：

```python
predict_future = pred_executor.submit(
    self.expert_predictor.predict, inps, self.model, i_layer, 1
)
```

当前实现里虽然用了线程池，但随后马上 `predict_future.result()`，所以逻辑上是“先发起预测，再收集预测结果”。它为后续调度提供 `self.predicted_next_demands`。

8. 计算 shared experts：

```python
shared_output = layer.mlp.shared_experts(inps)
```

9. flatten hidden states，准备专家执行：

```python
inps_flat = inps.view(-1, hidden_dim)
experts = layer.mlp.experts
```

10. 选择调度策略：

```python
if force_gpu:
    strategy = self.gpu_only_strategy
else:
    strategy = self.expert_strategy
```

这里 `force_gpu = is_prefill`，所以当前代码中：

- prefill 阶段强制使用 `GPUOnlyStrategy`。
- decode 阶段使用 `PrefetchHybridStrategy` 或配置下的混合策略。

11. 调用策略做决策：

```python
result_tuple = strategy.decide_and_prepare(
    i_layer,
    experts,
    selected_experts,
    routing_weights,
    self.n_expert,
    future_demands=self.predicted_next_demands,
    placement=self.placeholder_manager.snapshot(),
    is_prefill=is_prefill,
)
```

输入包括：

- 当前层编号。
- 当前层 experts。
- 当前 gate 选出的专家。
- 当前 routing weights。
- 当前 placeholder/GPU/CPU 驻留快照。
- 预测出的下一层专家需求。
- 当前阶段是 prefill 还是 decode。

输出包括：

- `cpu_experts`：当前层放到 CPU 执行的专家。
- `gpu_experts`：当前层放到 GPU/placeholder 执行的专家。
- `prefetch_experts`：下一层建议提前加载的专家。
- `expert_assignments`：每个当前专家对应的 token 和 routing weight。

12. 把调度结果包装成 `ExpertSchedule`：

```python
schedule = ExpertSchedule(
    cpu=[...],
    gpu=[...],
    preload=[...],
)
```

13. 把执行上下文包装成 `ExpertLayerContext`：

```python
context = ExpertLayerContext(
    layer=i_layer,
    experts=experts,
    inps_flat=inps_flat,
    hidden_dim=hidden_dim,
    assignments=build_assignments(expert_assignments),
)
```

14. 调用执行器：

```python
inps_after_experts = self.expert_executor.execute(schedule, context)
```

15. 合并 shared expert 输出和 routed expert 输出：

```python
total_expert_output = shared_output.view(-1, hidden_dim) + inps_after_experts
```

16. 加残差，恢复形状：

```python
inps = inps_residual + total_expert_output.reshape(batch_size, seq_len, hidden_dim)
```

17. 所有层跑完后，执行 final norm 和 lm head：

```python
inps = self.model.norm(inps)
lm_logis = self.lm_head(inps)
```

18. 返回 logits。

**调度调用链**
decode 阶段的主要调用链如下：

```text
deepseek.py
mixtral_forward()
  -> layer.mlp.gate(inps)
  -> GatePredictor.predict()
  -> placeholder_manager.snapshot()
  -> PrefetchHybridStrategy.decide_and_prepare()
       -> _build_expert_mask()
       -> _collect_active_experts()
       -> _organize_token_assignments()
       -> build_current_demands()
       -> unique_demands()
       -> ExpertLayerRequest(...)
       -> PDScopeScheduler.schedule()
            -> schedule_prefill() 或 schedule_decode()
            -> ExpertSchedule(...)
       -> 返回 cpu_expert_ids, gpu_expert_ids, preload_expert_ids, raw_assignments
  -> ExpertSchedule(...)
  -> ExpertLayerContext(...)
  -> ExpertExecutionManager.execute()
```

更细一点，`PrefetchHybridStrategy.decide_and_prepare()` 内部顺序是：

1. 根据 `selected_experts` 构建 expert mask。
2. 收集当前层活跃专家。
3. 组织 token 到 expert 的分配关系。
4. 构造 `current_demands`。
5. 接收或构造 `future_demands`。
6. 对 future demands 去重。
7. 获取或构造 `PlacementSnapshot`。
8. 更新专家命中统计。
9. 构造 `ExpertLayerRequest`。
10. 调用 `PDScopeScheduler.schedule()`。
11. 从 `ExpertSchedule` 取出 CPU/GPU/preload 专家 id。
12. 返回给 `deepseek.py`。

**Prefill 调度顺序**
`PDScopeScheduler.schedule_prefill()` 的核心顺序：

1. 取当前层需求：

```python
current = request.current
```

2. 把当前专家分成：

- `current_resident`：已经在 GPU 或 placeholder。
- `current_non_resident`：不在 GPU。

3. 过滤未来预测专家，得到：

- `future_non_resident`：未来可能用到、当前不在 GPU、且不在 loading 的专家。

4. 合并当前非驻留和未来非驻留：

```python
combined = current_non_resident + future_non_resident
```

5. 按 token 数和 score 排序。

6. 调用 `_select_global_queue()` 选出全局值得 GPU 化的候选队列。

7. 从全局队列中筛出当前层专家：

```python
current_global = [d for d in global_queue if d.source == "current"]
```

8. 调用 `_select_current_ondemand()` 决定哪些当前层专家值得按需加载到 GPU 执行。

9. 当前层专家最终被分成：

- `gpu`：已驻留或值得按需加载的当前专家。
- `cpu`：剩余当前专家。

10. 调用 `_select_preload()`，根据时间差和 placeholder 容量选择未来专家 preload。

11. 返回：

```python
ExpertSchedule(cpu=cpu, gpu=gpu, preload=preload, reason="prefill")
```

**Decode 调度顺序**
`PDScopeScheduler.schedule_decode()` 的核心顺序：

1. 如果当前层没有专家需求，返回空 schedule。

2. 计算当前活跃专家数 `k`。

3. 根据单 token CPU/GPU 延迟，计算理想 GPU 专家数 `n_g_rho`：

```python
n_g_rho = min(
    range(k + 1),
    key=lambda n_g: max(n_g * t_g, (k - n_g) * t_c),
)
```

4. 把当前专家分成：

- `current_resident`
- `current_non_resident`

5. 过滤未来非驻留专家 `future_non_resident`。

6. 计算下一层已经驻留的预测专家数量 `next_resident_count`。

7. 判断当前层和下一层是否低于理想 GPU 数：

```python
cur_below = len(current_resident) < n_g_rho
next_below = next_resident_count < n_g_rho
```

8. 根据四种情况决策：

- 当前不足、下一层也不足：fallback 到 prefill 策略，`reason="decode-fallback-prefill"`。
- 当前不足、下一层充足：补当前层 GPU 专家，`reason="decode-mode-a"`。
- 当前充足、下一层不足：当前层尽量用驻留 GPU，预加载下一层，`reason="decode-mode-b"`。
- 当前充足、下一层也充足：不额外 preload，`reason="decode-mode-c"`。

**执行调用链**
专家执行链如下：

```text
deepseek.py
mixtral_forward()
  -> ExpertExecutionManager.execute(schedule, context)
       -> 并行 execute_gpu_experts(...)
       -> 并行 execute_cpu_experts(...)
       -> 并行 preload_experts(...)
       -> 合并 GPU result 和 CPU result
       -> 返回 inps_after_experts
  -> shared_output + inps_after_experts
  -> residual add
```

`ExpertExecutionManager.execute()` 里最多启动三个 future：

```python
if schedule.gpu_expert_ids:
    execute_gpu_experts()

if schedule.cpu_expert_ids:
    execute_cpu_experts()

if schedule.preload:
    preload_experts()
```

这体现了本次重构的重要变化：专家计算和下一层预加载被拆开，并且可以并行推进。

**Placeholder 调用关系**
placeholder 相关调用主要有三类来源：

1. 初始化阶段：

```text
deepseek.__init__()
  -> ExpertPlaceholderManager(...)
  -> bring_expert_to_gpu()
       -> mark_static_gpu_resident()
       -> mark_cpu_resident()
```

2. 调度阶段：

```text
deepseek.mixtral_forward()
  -> placeholder_manager.snapshot()
  -> PrefetchHybridStrategy.decide_and_prepare()
  -> PDScopeScheduler.schedule()
```

调度器只看 snapshot，不直接修改 placeholder 状态。

3. 执行阶段：

```text
ExpertExecutionManager.execute_gpu_experts()
  -> get_placeholder_for_expert()
  -> acquire_placeholder()
  -> load_weights()
  -> release_by_layer()

ExpertExecutionManager.preload_experts()
  -> is_on_gpu()
  -> is_loading()
  -> mark_loading()
  -> acquire_free_placeholder()
  -> load_weights()
  -> unmark_loading()
```

也就是说：

- 调度只负责“决定”。
- 执行才负责“加载、占用、释放”。
- placeholder manager 是唯一维护驻留状态的模块。

**预测调用关系**
预测链如下：

```text
deepseek.mixtral_forward()
  -> GatePredictor.predict(inps, model, current_layer, lookahead=1)
       -> next_layer = model.layers[current_layer + 1]
       -> next_layer.mlp.gate(hidden_states)
       -> build_future_demands(next_layer_idx, predicted_experts, routing_weights)
       -> List[ExpertDemand]
  -> PrefetchHybridStrategy.decide_and_prepare(future_demands=...)
```

预测结果不会直接加载专家，而是先进入调度器，由调度器结合 placement、latency 和 placeholder capacity 决定是否 preload。

**数据流顺序**
一次 MoE 层的数据流可以概括为：

```text
hidden states
  -> 当前层 gate
  -> selected_experts + routing_weights
  -> expert_scheduling 整理 token assignment
  -> current demands
  -> expert_predictor 生成 future demands
  -> placeholder snapshot
  -> PDScopeScheduler 生成 ExpertSchedule
  -> ExpertExecutionManager 执行 CPU/GPU/preload
  -> routed expert output
  -> shared expert output 相加
  -> residual
  -> 下一层
```

其中几个关键数据对象的流转是：

```text
selected_experts, routing_weights
  -> raw_assignments
  -> build_assignments()
  -> ExpertLayerContext.assignments
  -> execute_gpu_experts() / execute_cpu_experts()
```

```text
predicted_experts, predicted_weights
  -> build_future_demands()
  -> future_demands
  -> ExpertLayerRequest.future
  -> PDScopeScheduler
  -> ExpertSchedule.preload
  -> ExpertExecutionManager.preload_experts()
```

```text
placeholder_manager internal state
  -> snapshot()
  -> PlacementSnapshot
  -> scheduler decision
  -> execute/preload modifies placeholder_manager
```

**重构后的优点**
这次重构主要改善了以下几点：

- 职责分离：`deepseek.py` 不再直接塞满专家调度、placeholder、执行细节。
- 调度可替换：`ExpertScheduler` 和 `PDScopeScheduler` 抽象出来后，后续可以新增策略而不改推理主流程。
- 执行可测试：`ExpertExecutionManager` 独立后，CPU/GPU/preload 执行路径可以单独测试。
- 数据结构清晰：通过 `ExpertDemand`、`ExpertSchedule`、`PlacementSnapshot` 等对象减少 tuple 语义混乱。
- placeholder 状态统一：专家是否在 GPU、是否 loading、placeholder 是否空闲都集中到 `ExpertPlaceholderManager`。
- 支持预测预取：`GatePredictor` 输出未来需求，调度器决定 preload，执行器实际加载。
- 延迟模型集中：调度策略不直接依赖 benchmark JSON，而是通过 `ExpertLatencyModel` 查询成本。

**需要注意的地方**
`deepseek.py` 末尾仍保留了一些旧方法：

- `_prefetch_next_layer_experts()`
- `run_expert_at_cpu()`
- `_execute_gpu_experts()`
- `_execute_cpu_experts()`

从当前主流程看，新的执行路径已经走 `ExpertExecutionManager.execute()`，这些旧方法目前更像是重构后的遗留兼容代码或待清理代码。后续如果确认没有外部调用，可以继续删除，进一步收敛 `deepseek.py` 的职责。

另外，当前 `mixtral_forward()` 中预测下一层专家时虽然用了 `ThreadPoolExecutor`，但马上调用了 `predict_future.result()`，因此实际并行收益有限。真正并行的是 `ExpertExecutionManager.execute()` 中 CPU/GPU/preload 三路执行。

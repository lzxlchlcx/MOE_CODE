# 性能分析报告（2026.4.15）

## 1. Microbench 测量数据准确性分析

### 当前测量结果
- **e（传输时间）**: 1.09ms
- **tg（GPU计算时间）**: 0.14ms
- **cpu_time_table**: 前6项通过线性插值填充，第7项开始为真实测量值

### 数据准确性问题

#### **问题1：单位转换错误（严重bug）**
- **现象**: 
  - microbench测量: token=1时CPU计算时间 = 0.074ms
  - JSON存储: cpu_time_table[0] = 0.074021（数值相同）
  - scheduler使用: 认为 cpu_time_table[0] = 0.074秒 = 74ms
- **根因**: 
  - `microbench.py:282`: `cpu_time_table_list = [float(t * 1000) for t in cpu_times_full]`
  - cpu_times_full是秒，乘1000转换为毫秒
  - 但JSON存储时没有明确单位，scheduler按秒使用
- **影响**: 
  - scheduler认为CPU计算需要74ms，远大于搬运成本e=1.09ms
  - 实际CPU只需0.074ms << 1.09ms搬运成本
  - 导致ondemand决策过度激进，大量不该搬运的专家被搬到GPU
- **实测验证**: 
  - 真实CPU计算时间（模拟测试）: token=1时0.34ms，token=4时0.67ms
  - 与microbench测量值（0.074ms、0.296ms）接近
  - 实测CPU专家平均6.30ms包含数据搬运开销

#### 问题2：前6项数据不可靠
- **现象**: `cpu_time_table[0~5] = 0.07~0.44ms`（线性插值）
- **根因**: microbench 使用 `window_size=7` 的卷积平滑，前6个token的测量数据被填0
- **影响**: batch_size=4 decode 时，每个专家处理 1-4 个token，决策依赖的前6项数据是估算值而非实测值

#### 问题3：测量环境与实际运行环境不一致
- **现象**: microbench 测量 `cpu_time_table` 时使用的是**单个专家预热后的稳定值**
- **实际**: 每层可能有多个CPU专家并发执行，存在线程竞争和内存带宽瓶颈
- **影响**: 实际CPU计算时间可能高于测量值（实测 6.30ms vs 表中 0.30ms）

#### 问题4：实测值包含额外开销
- **实测 expert_compute-cpu**: 平均6.30ms（2594次）
- **表中值**: token=4时0.30ms
- **差异原因**:
  1. 实测包含GPU->CPU->GPU数据搬运开销（约1-2ms）
  2. 多个CPU专家并发竞争（CPU核心、内存带宽、Cache）
  3. CPU专家可能处理更多token（>4个）
  4. 预热不充分

### 结论
**当前microbench数据存在严重单位转换错误**：
- ✗ **单位错误**: cpu_time_table存储的是毫秒，但scheduler按秒使用（偏差1000倍）
- ✗ `cpu_time_table` 前6项是估算值，不可靠
- ✓ 实测CPU时间（6.30ms）包含数据搬运+并发竞争，是真实开销
- ✗ 单位错误导致ondemand决策过度激进

## 2. 预取调度策略有效性分析

### 当前预取效果
- **prefetch占比**: 3.8%（2773次命中）
- **命中率**: 69.8%（GPU常驻专家）
- **ondemand占比**: 6.2%
- **CPU占比**: 20.2%

### 预取效果不显著的原因

#### 原因1：热点专家已在GPU常驻
- GPU上放置了 830/1664 = 49.9% 的专家
- 热点策略优先将高频专家放在GPU
- 预取检查 `is_expert_in_gpu_now`，跳过已在GPU的专家
- **结果**: 只有CPU上的冷门专家被预取，但它们通常不是热点

#### 原因2：预取预测准确率有限
- 预取基于**下一层可能需要的专家**预测
- 但MoE路由具有动态性，预测不一定准确
- 预取了但未被使用的专家会浪费带宽

#### 原因3：异步CUDA Stream效果有限
- 预取改为CUDA Stream异步执行
- 但预取数量太少（3.8%），对整体性能影响有限
- 预取的专家可能已被ondemand机制覆盖

### 结论
**当前预取策略发挥了作用，但效果有限**：
- ✓ 技术实现正确（CUDA Stream异步）
- ✗ 预取占比太低（3.8%），对吞吐影响微弱
- ✗ 热点专家已在GPU，预取目标空间有限

## 3. CPU-GPU负载均衡分析

### 当前负载分布（batch_size=4，平衡ondemand策略）

| 指标 | 数值 | 占比 |
|------|------|------|
| GPU总工作时间 | 109997ms | 91.7% |
| CPU总工作时间 | 84499ms | 70.4% |
| 并行总时间 | 119963ms | 100% |
| GPU空闲时间 | 9966ms | 8.3% |
| CPU空闲时间 | 35463ms | 29.6% |

### 负载均衡评估

**GPU/CPU负载比**: 1.30:1（GPU过载）

**负载不均衡的表现**：
- CPU空闲29.6%，等待GPU完成
- GPU空闲8.3%，几乎满负载运行
- GPU成为串行瓶颈

**并行度对比**：
- 实际并行度: 1.62x
- 理论最优并行度: 1.77x（假设GPU和CPU完美并行）
- 性能损失: 8.5%

### 负载不均衡的根因

#### 原因1：ondemand搬运增加GPU负担
- ondemand专家数量: 6.2%（4548次）
- 每次搬运成本: ~1.09ms（同步执行在GPU线程内）
- 总搬运开销: 4548 × 1.09ms ≈ **5秒**加到GPU线程
- CPU工作量反而减少（专家被搬到GPU了）

#### 原因2：GPU处理更多专家
- GPU专家占比: 70% + 6.2%ondemand + 3.8%prefetch ≈ 80%
- CPU专家占比: 20%
- GPU线程工作量 > CPU线程工作量

#### 原因3：ondemand搬运是同步阻塞的
- `_async_ondemand` 实际上是同步的（deepseek.py:380-417）
- 搬运过程阻塞GPU线程，CPU线程快速完成后等待GPU

### 结论
**当前负载不均衡，GPU过载**：
- ✗ GPU/CPU负载比 1.30:1，GPU成为瓶颈
- ✗ CPU空闲29.6%，资源浪费
- ✗ 需进一步优化让CPU工作量≈GPU工作量

## 4. 当前瓶颈分析

### 系统瓶颈定位

#### 瓶颈1：GPU线程同步阻塞（严重）
**位置**: `deepseek.py:380-417` `_async_ondemand`

**现象**:
- ondemand搬运在GPU线程内同步执行
- 每次搬运约1.09ms，4548次搬运累计约5秒
- GPU线程承担：GPU专家计算 + ondemand搬运 + prefetch

**影响**:
- GPU线程工作量增加 → CPU线程空闲等待
- 搬运开销抵消了GPU计算优势

**优化方向**:
- 让ondemand搬运真正异步（不在GPU线程内阻塞）
- 或减少ondemand数量，保持负载均衡

#### 瓶颈2：CPU计算时间表不准确（中等）
**位置**: `microbench.py` cpu_time_table测量

**现象**:
- 前6项是线性插值估算值（不可靠）
- 实测CPU时间（10.24ms）远高于表中值（1-2ms）

**影响**:
- ondemand决策依赖不准确的数据
- 可能做出错误的搬运决策

**优化方向**:
- 改进microbench，直接测量1-6个token的CPU时间
- 或减小平滑窗口大小

#### 琶颈3：热点策略占用GPU过多（设计问题）
**位置**: GPU常驻专家选择策略

**现象**:
- GPU上放置830个专家（49.9%）
- 热点专家优先放GPU
- 导致大部分专家命中GPU，CPU无工作可做

**影响**:
- batch_size=1时命中率86%，并行度低（CPU空闲）
- batch_size=4时命中率70%，负载仍不均衡

**优化方向**:
- 动态调整GPU常驻专家数量
- 根据batch_size调整热点策略
- 减少GPU常驻专家，让更多专家走CPU（提升并行度）

#### 瓶颈4：placeholder数量有限（架构限制）
**位置**: 占位专家数量固定为6个

**现象**:
- 只有6个placeholder可用于ondemand搬运
- 当ondemand专家>6时，部分专家无法搬运

**影响**:
- 限制了ondemand的并行度
- 多个ondemand专家串行处理

**优化方向**:
- 增加placeholder数量（需更多显存）
- 或动态分配placeholder

### 瓶颈优先级排序

| 优先级 | 瓶颈 | 影响 | 优化难度 |
|--------|------|------|----------|
| **P0** | GPU线程同步阻塞 | 严重（GPU过载） | 中等 |
| **P1** | CPU时间表不准确 | 中等（决策错误） | 低 |
| **P2** | 热点策略占用GPU过多 | 中等（负载不均衡） | 高 |
| **P3** | placeholder数量有限 | 低（架构限制） | 高 |

### 当前性能瓶颈总结

**主要瓶颈**: GPU线程过载，CPU空闲等待
- GPU承担了70%专家计算 + 6.2%ondemand搬运开销
- CPU只承担20%专家计算，大量空闲
- 并行度1.62x，理论最优1.77x，损失8.5%

**次要瓶颈**:
- CPU时间表前6项估算不准确
- 热点策略让过多专家在GPU（降低并行度）
- placeholder数量限制ondemand并发

**优化目标**: 让GPU时间 ≈ CPU时间，达到负载均衡

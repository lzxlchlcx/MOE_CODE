# PDScope 复现计划

## 目标

根据 PDScope 论文，在现有 MoE 推理框架上实现阶段感知的调度和系统级优化。分阶段渐进实施，先做调度改进（学术价值高），再做系统级优化（工程收益大）。

## 整体路线

```
Phase 1: 基类抽取 + 统一架构
Phase 2: AdaptSched Decode 负载均衡调度器
Phase 3: AdaptSched Prefill 三步法调度器
Phase 4: 多流并发传输（3 CUDA Stream）
Phase 5: CPU 缓存预取 + 后台线程隔离
Phase 6: 集成测试 + microbench 更新
```

## Phase 2: AdaptSched Decode 负载均衡调度器

**目标**：在 `scheduler.py` 中实现论文 §3.2 的 Decode 阶段显式负载均衡。

### 2.1 核心算法

Decode 阶段每个专家只处理 1 个 Token，不存在"大小专家"之分。调度目标：

```
n_g^ρ = argmin max(n_g^ρ · tg, (k - n_g^ρ) · tc)
```

即找到 GPU/CPU 同时完工的最优专家分配数。

根据当前层和下一层的 GPU 驻留专家数与 `n_g^ρ` 的关系，触发三种模式：

| 模式 | 条件 | 动作 |
|---|---|---|
| A (保当前层) | 当前层 < n_g^ρ，下一层 >= n_g^ρ | 暂停预取，全力按需加载当前层 |
| B (保下一层预取) | 当前层 > n_g^ρ，下一层 < n_g^ρ | 卸载当前层多余专家，全力预取下层 |
| C (双超标清理) | 两层都 > n_g^ρ | 只卸载，释放带宽 |

### 2.2 实现细节

在 `scheduler.py` 的 `ExpertScheduler` 中添加：

```python
def decide_decode_schedule(self, layer_idx, selected_experts, 
                           n_experts_on_gpu_current, n_experts_on_gpu_next,
                           total_activated_experts) -> DecodeSchedule:
    """Decode阶段负载均衡调度
    
    Returns:
        DecodeSchedule: 包含
            - ondemand_experts: 需要ondemand加载的专家
            - prefetch_experts: 可以预取的下一层专家
            - mode: A/B/C
            - n_g_optimal: 最优GPU专家数
    """
```

需要新增的参数（来自 microbench）：
- `tc`: CPU 计算单个专家的时间（decode 阶段 token=1 时的 CPU 时间表值）
- 已有的 `tg` (GPU 计算时间), `e` (传输时间)

### 2.3 集成到基类

在 `base_model.py` 的 `_dispatch_experts()` 中，Decode 阶段调用新的调度方法：

## Phase 3: AdaptSched Prefill 三步法调度器

**目标**：实现论文 §3.2 的 Prefill 阶段三步调度法。

### 3.1 Step 1: 全局排位（跨层成本建模）

将当前层 L 和预测的下层 L+1 的专家放入同一个池子，按 Token 数量升序排列。对每个候选边界 i 计算全局成本：

```
T_all^G = α + (n + n' - i + 1) · t_io + t_g    # GPU总成本
T_all^C = t_c(E_all[0:i]) + t_attn               # CPU总成本
```

当 `T_all^G < T_all^C` 时，边界右侧专家入选全局队列。

### 3.2 Step 2: 局部重排

从全局队列中仅筛选当前层专家，确保当前层不被"饿死"。

### 3.3 Step 3: 置信度感知预取

计算 I/O 气泡中的期望效用 ξ：
```
ξ = R_hit · (f - ⌊f⌋ + 1) · t_io - (1 - R_hit) · (⌊f⌋ - f) · t_io
```

仅当 ξ > 0 时执行预取。

                             next_layer_predicted_experts,

## Phase 4: 多流并发传输

**目标**：将专家权重的 3 个矩阵（gate/up/down）使用 3 个独立 CUDA Stream 并发传输，提升 PCIe 带宽利用率。

### 4.1 修改 `_async_ondemand` 和 `_async_load_expert`

当前代码（单流顺序）：
```python
with torch.cuda.stream(self._prefetch_stream):
    for name in ["gate_proj", "up_proj", "down_proj"]:
        dst.copy_(src, non_blocking=True)  # 顺序拷贝
```

改为（三流并发）：
```python
param_names = self._get_expert_param_names()
streams = [torch.cuda.Stream() for _ in param_names]

for stream, name in zip(streams, param_names):
    with torch.cuda.stream(stream):
        dst = getattr(target_placeholder, name).weight.data
        src = getattr(self._get_expert(layer_idx, expert_id), name).weight.data
        dst.copy_(src, non_blocking=True)

# 等待所有流完成
for stream in streams:
    torch.cuda.synchronize(stream)
```

### 4.2 流的生命周期管理

- 在基类 `__init__` 中预创建 3 个常驻 CUDA Stream（而非每次临时创建）
- 命名：`self._transfer_streams = [torch.cuda.Stream() for _ in range(3)]`
- 考虑额外 1 个流用于预取（总计 4 个流）

### 4.3 预期效果

论文报告 PCIe 带宽利用率从 ~30% 提升到 66%–80%。

### 5.2 后台调度线程物理隔离

```python
import os

# 主线程绑定到计算核心
os.sched_setaffinity(0, set(range(16)))

# 后台调度线程绑定到独立核心
def _schedule_thread_func():
    os.sched_setaffinity(0, {31})  # 绑定到核心31
    while not stop_event.is_set():
        task = ring_buffer.get()  # 无锁环形缓冲区
        schedule = compute_schedule(task)
        result_buffer.put(schedule)
```

### 5.3 无锁环形缓冲区

使用 `collections.deque`（线程安全）或 `queue.SimpleQueue` 作为轻量级替代：
```python
from collections import deque

class LockFreeRingBuffer:
    def __init__(self, capacity):
        self.buffer = deque(maxlen=capacity)
        self._lock = threading.Lock()  # 先用锁，后续优化为原子操作
    
    def put(self, item):
        with self._lock:
            self.buffer.append(item)
    
    def get(self):
        with self._lock:
            return self.buffer.popleft() if self.buffer else None
```

## 关键设计决策

1. **预测器**：暂用 Gate 直推（将当前层 inps 送入下一层 Gate），不训练 PhasePer
2. **调度器**：先实现 Decode 负载均衡（更有学术价值，算法清晰），再实现 Prefill 三步法
3. **系统优化**：渐进式，先验证调度效果，再加多流并发、缓存预取
4. **统一性**：通过抽象基类统一四个模型，所有新特性只需在基类实现一次
5. **参数来源**：所有 e/tg/tc/cpu_time_table 均来自 microbench 实测，不硬编码

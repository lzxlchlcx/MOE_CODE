# Plan: 系统级协同优化引擎（"最后一公里"）

## 目标
实现 PDScope 论文 Section 3.3 的系统级执行优化，消除运行时开销。

## 现状分析

| 开销源 | 位置 | 影响 |
|--------|------|------|
| `pin_memory()` 每次传输时调用 | `deepseek.py:399-402` | DMA 无法立即启动，先做一次内存拷贝 |
| 每层创建/销毁 6+ Python 线程 | `deepseek.py:1670-1677` | 线程创建开销 ~0.1ms/个 × 26 层 |
| `torch.cuda.synchronize()` 全局同步 | `deepseek.py:1482,1527,1852` | 阻塞所有 GPU 操作，而非只等特定操作 |
| prefetch 线程每层 join+重建 | `deepseek.py:1357-1367` | 线程无法跨层存活，丢失预取时间窗口 |
| `_transfer_streams` 无并发保护 | `_async_load_expert` | prefetch 线程和主线程可能同时使用同一个 stream |

## 优化方案（4 项）

### S1: 预锁页所有专家权重（Pre-pin）
**优先级: 高 | 改动: ~20 行 | 风险: 低**

在 `__init__` 中加载模型后、设置专家位置前，一次性将所有专家权重 `pin_memory()`。

```python
# __init__ 中，set_expert_loc 之后
for layer in self.model.layers:
    for expert in layer.mlp.experts:
        for name in ["gate_proj", "up_proj", "down_proj"]:
            w = getattr(expert, name, None)
            if w is not None and not w.weight.data.is_pinned():
                w.weight.data = w.weight.data.pin_memory()
```

然后 `_async_load_expert` 中删除 `pin_memory()` 调用（line 399-402），直接启动 DMA。

**收益**: 消除每次传输 ~0.05-0.1ms 的 pin_memory 开销（26 层 × 多个专家 × 每次推理）。

### S2: 线程池替代逐层线程创建
**优先级: 中 | 改动: ~30 行 | 风险: 低**

用 `concurrent.futures.ThreadPoolExecutor` 替代 `threading.Thread`：

```python
# __init__ 中
self._executor = ThreadPoolExecutor(max_workers=6)

# Section 11 中
gpu_future = self._executor.submit(process_gpu_experts)
cpu_future = self._executor.submit(process_cpu_experts)
gpu_results = gpu_future.result()
cpu_results = cpu_future.result()
```

**收益**: 消除 26 层 × 6+ 线程的创建/销毁开销。

### S3: 事件同步替代全局同步
**优先级: 高 | 改动: ~25 行 | 风险: 中**

将 `torch.cuda.synchronize()` 全局同步替换为 CUDA Event 同步：

```python
# __init__ 中：为每个 placeholder 创建一个 event
self._transfer_events = [torch.cuda.Event() for _ in range(6)]

# _async_load_expert 末尾：记录 event 而非同步
for s in self._transfer_streams:
    s.record_event(self._transfer_events[ph_idx])

# process_experts_in_placeholder 中：只等特定 event
self._transfer_events[ph_idx].synchronize()
```

同样，`run_expert_at_gpu` (line 1852) 和 `run_expert_at_cpu` (line 1900) 中的 `torch.cuda.synchronize()` 改为 event-based。

**收益**: 消除全局同步导致的其他 GPU 操作被不必要阻塞。Decode 阶段每个 expert 只处理 1 token，全局同步开销占比高。

### S4: 持久化预取线程 + 任务队列
**优先级: 中 | 改动: ~40 行 | 风险: 中**

将每层 join+重建的 prefetch 线程改为持久化线程 + `queue.Queue`：

```python
# __init__ 中
self._prefetch_queue = queue.Queue()
self._prefetch_done = threading.Event()

def _prefetch_worker():
    while not self._prefetch_done.is_set():
        try:
            tasks, layer_idx = self._prefetch_queue.get(timeout=0.01)
            for eid in tasks:
                self._async_load_expert(layer_idx, eid)
                # ... 更新 prefetch_list
        except queue.Empty:
            pass

self._prefetch_worker_thread = Thread(target=_prefetch_worker, daemon=True)
self._prefetch_worker_thread.start()
```

Section 10 中改为 `self._prefetch_queue.put((tasks, layer_idx))`，无需 join。

**收益**: 
- 消除线程创建开销
- 预取线程在层间保持存活，下一层计算时立即开始预取下下层
- 更好的跨层 pipelining

## 不实施的项目

| 项目 | 原因 |
|------|------|
| CUDA Graph | 需要固定操作序列和形状，但专家数量和分配每层动态变化；PyTorch CUDA Graph 对动态控制流支持有限 |
| CPU L3 缓存预取 | Python 无法直接调用 `_mm_prefetch`，需要 C 扩展；收益难以在 Python 层面验证 |
| 线程物理核心绑定 | `os.sched_setaffinity` 在容器/虚拟环境中可能不可用；收益依赖硬件拓扑 |

## 实施范围：S1 + S2

S3（事件同步）和 S4（持久化预取线程）留后续验证。

# 2026.3.29

## 目标
测试DeepSeek模型在当前框架下的性能，发现吞吐低、命中率为0、GPU/CPU负载不均衡

## 执行
执行 `cd opensource && python latency.py --model "/mnt/g/Models/DeepSeek-v2-lite-chat" --batch_size 1`

## 结果
1. **吞吐**: ~1.5-2.26 token/s
2. **命中率**: 0 (代码bug)
3. **GPU/CPU负载不均衡**: expert_compute(GPU) 0.42ms vs expert_compute-cpu 5.38ms
4. **GPU专家数量过少**: 由于calc_n_expert_on_gpu中硬编码的 `-930` 导致计算错误

## 反思
发现两个问题：
1. **calc_n_expert_on_gpu函数bug**: 硬编码magic number导致GPU专家数量计算错误
2. **命中率统计bug**: cnt_expert_hit从未被增加，命中率永远为0

### 修复1: calc_n_expert_on_gpu
修改 `deepseek.py` 第566行附近：
- 移除硬编码的 `-930`
- 添加buffer_factor=0.7
- 添加详细调试信息

**修复后**:
- GPU专家数量从几十提升到 **830/1664**
- expert_compute次数从1918次增加到 **4106次** (+114%)
- expert_compute-cpu次数从3701次降到 **1570次** (-58%)

### 修复2: 命中率统计
修改 `deepseek.py` 第670行和第1193行附近：
- 初始化 `cnt_expert_all = 0`
- 在循环中正确统计命中和未命中专家数
- 添加除零保护

**修复后**:
- 命中率从 0 变为 **0.68-0.81** (平均0.74)
- 说明74%的路由专家在GPU上（热点策略生效）

## 下一步
- 测试模型在其他框架下的性能，VLLM、llamacpp
- 输出乱码问题(涉及、prose重复)是DeepSeek-V2-Lite模型本身问题，非代码问题

# 2026.3.31

## 目标
测试 DeepSeek 模型在 vLLM 框架下的性能，实现 GPU 显存不足时的 CPU 卸载

## 预期
vLLM 框架性能应该比当前 scope 框架更好，即使在 CPU 卸载情况下也能达到或超过 2.26 token/s

## 执行
1. 创建 vLLM 测试脚本 `opensource/vllm_benchmark.py`
2. 添加内存优化参数：`--cpu_offload_gb`、`--enforce_eager`、`--max_model_len`
3. 执行测试命令：
   ```bash
   cd opensource
   conda activate vllm
   python vllm_benchmark.py \
     --model "/mnt/g/Models/DeepSeek-v2-lite-chat" \
     --batch_size 1 \
     --cpu_offload_gb 20 \
     --max_model_len 2048 \
     --enforce_eager
   ```

## 结果
| 指标 | 数值 |
|------|------|
| 框架 | vLLM |
| 模型 | DeepSeek-V2-lite |
| Batch size | 1 |
| Input tokens | 128 |
| Output tokens | 32 |
| **CPU offload** | 20.51GB |
| **GPU显存使用** | 9.0 GiB (模型加载) |
| 总吞吐 | 2.60 token/s |
| KV cache | 333,296 tokens |

**内存分布说明：**
- 模型总参数量：约 29.51GB
- GPU显存：9.0 GiB (约30.5%)
- CPU内存：20.51GB (约69.5%)

**GPU显存详细占用分析：**
1. **模型参数**: 约 4.5-5.5 GiB (MoE的共享参数、部分专家层)
2. **KV Cache**: 约 3.0-3.5 GiB (333,296 tokens × 每个token的KV缓存)
3. **激活值/临时张量**: 约 0.5-1.0 GiB (前向/后向传播的中间结果)
4. **vLLM运行时开销**: 约 0.5-1.0 GiB (CUDA context、显存管理等)

**总计**: ~9.0 GiB

### 与 scope 框架对比
- scope 框架：2.26 token/s
- vLLM 框架：2.60 token/s
- 性能提升：+15%

## 反思
vLLM 框架即使在 CPU 卸载情况下也比 scope 框架性能更好（+15%），说明：
1. vLLM 的 PagedAttention 和高效执行引擎确实有性能优势
2. CPU 卸载机制工作正常，没有明显拖慢性能
3. 需要进一步测试更大的 batch size 来充分挖掘 vLLM 的性能潜力

## 目标
尝试减少 vLLM 的 CPU offload 比例，观察能否在 GPU 显存容量允许的范围内提升性能

## 预期
降低 `cpu_offload_gb` 值，将更多参数放在 GPU 上，减少 CPU-GPU 数据传输开销，从而提升吞吐

## 执行
1. 测试 `cpu_offload_gb=10` 配合 `max_model_len=256` 和 `kv_cache_dtype=fp8`
2. 发现 DeepSeek 的 MLA 注意力机制不支持 FP8 KV cache
3. 测试 `cpu_offload_gb=10` 配合 `max_model_len=256`（不使用 FP8）

## 结果
1. **FP8 兼容性问题**：DeepSeek-V2-lite 使用 MLA (Multi-Head Latent Attention) 注意力机制，vLLM 的所有可用注意力后端（FLASH_ATTN_MLA、FLASHMLA、TRITON_MLA）都不支持 FP8 KV cache

2. **CPU offload 下限测试**：
   - `cpu_offload_gb=20`：✓ 成功，模型加载占用 9.0 GiB，KV cache 可用 333,296 tokens
   - `cpu_offload_gb=10`：✗ 失败，模型加载占用 19.48 GiB，几乎占满 24 GiB GPU 显存，没有剩余空间给 KV cache

3. **GPU 显存使用分析**：
   - 模型总参数量：约 29.51GB
   - GPU 显存容量：24 GiB
   - 模型加载后剩余显存：约 4-5 GiB（需考虑 CUDA context、激活值等）
   - 安全的 GPU 参数存储量：约 8-9 GiB（剩余空间用于 KV cache 和激活值）
   - 因此 CPU offload 下限约为 20-21 GB

## 反思
1. **DeepSeek 特殊架构**：MLA 注意力机制导致无法使用 FP8 压缩 KV cache，限制了内存优化空间
2. **GPU 显存预算**：对于 24 GiB 显存，模型参数需要留出足够空间给 KV cache 和激活值，不能一味追求减少 CPU offload
3. **当前配置是最优**：`cpu_offload_gb=20` 是在当前硬件条件下的最佳平衡点

### 下一步
- 测试 batch_size=4/8/16 时的性能
- 测试其他模型（Qwen、Mixtral、Moon）在 vLLM 下的性能
- 对比 scope 框架和 vLLM 在更大 batch size 下的性能差异

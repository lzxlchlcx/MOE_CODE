## 目标
尝试减少 vLLM 的 CPU offload 比例，观察能否在 GPU 显存容量允许的范围内提升性能

## 预期
降低 `cpu_offload_gb` 值，将更多参数放在 GPU 上，减少 CPU-GPU 数据传输开销，从而提升吞吐

## 执行
1. 测试 `cpu_offload_gb=10` 配合 `max_model_len=256` 和 `kv_cache_dtype=fp8`
2. 发现 DeepSeek 的 MLA 注意力机制不支持 FP8 KV cache
3. 测试 `cpu_offload_gb=10` 配合 `max_model_len=256`（不使用 FP8）

## 结果
1. **FP8 兼容性问题**：DeepSeek-V2-lite 使用 MLA (Multi-Head Latent Attention) 注意力机制，vLLM 的所有可用注意力后端（FLASH_ATTN_MLA、FLASHMLA、TRITON_MLA）都不支持 FP8 KV cache

2. **CPU offload 下限测试**：
   - `cpu_offload_gb=20`：✓ 成功，模型加载占用 9.0 GiB，KV cache 可用 333,296 tokens
   - `cpu_offload_gb=10`：✗ 失败，模型加载占用 19.48 GiB，几乎占满 24 GiB GPU 显存，没有剩余空间给 KV cache

3. **GPU 显存使用分析**：
   - 模型总参数量：约 29.51GB
   - GPU 显存容量：24 GiB
   - 模型加载后剩余显存：约 4-5 GiB（需考虑 CUDA context、激活值等）
   - 安全的 GPU 参数存储量：约 8-9 GiB（剩余空间用于 KV cache 和激活值）
   - 因此 CPU offload 下限约为 20-21 GB

## 反思
1. **DeepSeek 特殊架构**：MLA 注意力机制导致无法使用 FP8 压缩 KV cache，限制了内存优化空间
2. **GPU 显存预算**：对于 24 GiB 显存，模型参数需要留出足够空间给 KV cache 和激活值，不能一味追求减少 CPU offload
3. **当前配置是最优**：`cpu_offload_gb=20` 是在当前硬件条件下的最佳平衡点

### 下一步
- 测试 batch_size=4/8/16 时的性能
- 测试其他模型（Qwen、Mixtral、Moon）在 vLLM 下的性能
- 对比 scope 框架和 vLLM 在更大 batch size 下的性能差异
- 尝试使用多卡并行测试（tensor_parallel_size > 1）

### 发现的 Bug

#### Bug1: Logger 未接入（严重）
- `log_expert_hit()`、`log_layer_stats()`、`log_expert_class()` 在 deepseek.py 中从未调用
- Logger 报告命中率 0%、并行度 1.0x，与实际不符

- `process_experts_in_placeholder` 调用 `run_expert_at_gpu(12, 1, ...)` → 永远执行层12专家1
- `process_experts_loading` 调用 `run_expert_at_gpu(2, 3, ...)` → 永远执行层2专家3
- **根本原因**: 应该用 `expert_to_placeholder` 映射找到占位专家对象直接执行，而非通过 `run_expert_at_gpu` 查找原始模型专家
- **修复位置**: `deepseek.py:1399-1427`, `deepseek.py:1429-1461` → 使用 `placeholder(expert_input)` 替代硬编码

#### Bug3: layer_time_accumulator 双重累加
- 第1759行和第1783行对 `layer_time_accumulator[i_layer]` 重复累加
- **修复位置**: 删除第一个累加点（第1759行）

#### Bug4: CPU 专家预热每次 decode 重复触发
- `process_cpu_experts()` 中每次遇到 CPU 专家都做 dummy warmup
- **修复位置**: 添加 `_cpu_expert_warmed_up` 标志，只预热一次

### 并行度分析
- 大部分层 CPU Time = 0ms（所有选中专家都在GPU上）→ 并行度 ~0.92x
- 仅少数层有 CPU 专家（层7、8、9、11、13、14）→ 并行度 1.34-1.88x
- **根因**: batch_size=1 时 top-6 专家大部分在 GPU 上，CPU 无工作可做
- 总体平均并行度被无 CPU 工作的层拉低到 1.35x

## 反思
1. **占位专家硬编码是最严重的 bug** — 导致推理结果完全错误（乱码问题可能与此相关，但当前乱码更可能是模型本身问题）

3. **命中率 86% 已接近理论上限** — GPU 上放了 1067/1664 = 64% 的专家，但 top-6 选择使命中率远高于 64%

### 修复6: cpu_time_table 前6项为0导致 ondemand 失效

- **问题**: decode 阶段 `if TG < TC` 且 `token_count > 1`，小 batch 下 CPU 计算成本低于搬运成本，导致 24.7% 的专家留在 CPU 上

## 性能分析总结（四个关键问题）

### 1. Microbench 测量数据准确性
**结论**: 部分准确，但存在偏差

- ✓ `e=1.09ms` 和 `tg=0.14ms` 的测量相对可靠
- ✗ `cpu_time_table` 前6项是线性插值估算值（不可靠）
- ✗ 实测CPU时间（10.24ms）远高于表中值（1-2ms），测量环境与实际运行环境不一致

**根因**: 
- window_size=7的卷积平滑导致前6项填0后插值
- microbench测量单专家预热后的稳定值，实际多专家并发存在竞争

详见：`docs/performance_analysis_2026-04-15.md`

### 2. 预取调度策略有效性
**结论**: 发挥了作用，但效果有限（占比仅3.8%）

- ✓ 技术实现正确（CUDA Stream异步）
- ✗ 预取占比太低，对吞吐影响微弱
- ✗ 热点专家已在GPU常驻，预取目标空间有限

**根因**: 
- GPU上放置830/1664=49.9%专家，热点策略优先高频专家
- 预取检查 `is_expert_in_gpu_now` 跳过已在GPU的专家
- 只有CPU上的冷门专家被预取，但它们通常不是热点

### 3. CPU-GPU负载均衡
**结论**: 负载不均衡，GPU过载（负载比1.30:1）

| 指标 | GPU | CPU |
|------|-----|-----|
| 总工作时间 | 109997ms (91.7%) | 84499ms (70.4%) |
| 空闲时间 | 9966ms (8.3%) | 35463ms (29.6%) |
| 负载比 | 1.30:1 | GPU过载 |

**并行度对比**: 
- 实际: 1.62x
- 理论最优: 1.77x
- 性能损失: 8.5%

**根因**: 
- ondemand搬运在GPU线程内同步执行（增加约5秒开销）
- GPU承担80%专家，CPU只承担20%
- CPU线程快速完成，等待GPU线程

### 4. 当前瓶颈定位
**主要瓶颈**: GPU线程过载，CPU空闲等待

| 优先级 | 瓶颈 | 影响 | 优化难度 |
|--------|------|------|----------|
| P0 | GPU线程同步阻塞（ondemand搬运） | GPU过载，CPU空闲29.6% | 中等 |
| P1 | CPU时间表不准确 | 决策可能错误 | 低 |
| P2 | 热点策略占用GPU过多 | 并行度低 | 高 |
| P3 | Placeholder数量限制（6个） | Ondemand并发受限 | 高 |

**优化方向**:
- P0: 让ondemand搬运真正异步（不在GPU线程内阻塞）
- P1: 改进microbench，直接测量1-6个token的CPU时间
- P2: 动态调整GPU常驻专家数量，让CPU工作量≈GPU工作量
- P3: 增加placeholder数量（需更多显存）

# 2026.4.20

## 目标
审查 microbench.py 的测量代码，分析 microdeepseek.txt 中 GPU 计算时间异常的原因

## 预期
GPU 计算时间应随 token 数线性增长，且 token=1 时 GPU 不应慢于 CPU

## 执行
1. 查看 `microdeepseek.txt` 数据，发现异常
2. 逐段审查 microbench.py 的 GPU/CPU/拷贝测量代码
3. 审查 `deepseek.py` 的 `run_expert_at_cpu` 实现

## 结果

### 异常数据
```
Token=1:  GPU=0.3040ms, CPU=0.0740ms  ← CPU 远快于 GPU（不合理）
Token=2:  GPU=0.1278ms, CPU=0.1480ms  ← GPU 突然下降 2.4 倍（不合理）
Token=3:  GPU=0.1118ms, CPU=0.2221ms  ← GPU 几乎不变（不合理）
Token=64: GPU=0.1421ms, CPU=5.8706ms  ← GPU 基本持平，CPU 线性增长
Transfer: 全部 1.0904ms（固定值，合理但偏高）
```

### 发现的问题

#### 问题1: GPU 测量只预热1次、只测量1次（核心问题）
- **位置**: `microbench.py:228-236`
- **现象**: token=1 时 GPU=0.304ms（包含 CUDA kernel JIT 编译），token=2 后骤降到 0.12ms（kernel 已缓存）
- **对比**: CPU 测量预热1次 + 测量20次取平均（`microbench.py:254-263`），GPU 只预热1次 + 测量1次
- **影响**: GPU 时间测量不准确，且包含启动开销

#### 问题2: `run_expert_at_cpu` 内部多余 `torch.cuda.synchronize()`
- **位置**: `deepseek.py:1988`
- **现象**: CPU 计算中调用了 `torch.cuda.synchronize()`，测量时间包含了 GPU 等待时间
- **影响**: CPU 时间可能偏高（包含 GPU 残留操作等待时间）

#### 问题3: 拷贝测量与计算测量使用不同专家
- **位置**: `microbench.py:178`（拷贝用专家2）vs `microbench.py:220`（GPU计算用专家1）
- **现象**: 拷贝测量用 `range(2,3)`（专家2），GPU/CPU 计算测量用专家1
- **影响**: pin_memory 只处理了专家2，专家1 在后续 GPU 测量中未经过 pin_memory

#### 问题4: Transfer Time 1.09ms 偏高
- DeepSeek-V2-Lite 单专家约 140MB，PCIe 带宽正常应 0.2-0.5ms
- 可能原因：PCIe 3.0 x8 或 pin_memory 未完全生效

### 对 ondemand 调度的影响

当前异常数据导致调度决策偏差：
```
当前数据（有问题）: e=1.09ms, tg≈0.12ms（avg）
  → TG = e + tg = 1.21ms，TC（小token）≈ 0.07ms
  → TG >> TC，ondemand 永远不搬运（错过搬运机会）

修复后（预期）: e≈0.3ms, tg≈0.05ms
  → TG = 0.35ms，TC（token=10）≈ 0.74ms
  → TG < TC，开始搬运（GPU 优势显现）
```

核心影响：tg 被高估（0.12ms vs 实际可能 0.05ms），ondemand 算法**高估 GPU 计算成本**，错过搬运机会。

## 反思
1. **GPU 测量方法不一致是最根本的问题** — CPU 测 20 次取平均，GPU 只测 1 次，导致数据不可比
2. **CUDA kernel JIT 编译是 GPU 时间骤降的原因** — token=1 包含编译开销，后续 kernel 缓存后时间骤降
3. **需要统一测量方法** — GPU 也应多次预热 + 多次测量取平均

## 修复建议
1. GPU 测量增加预热到 10 次，测量增加到 20 次取平均
2. `run_expert_at_cpu` 删除 `torch.cuda.synchronize()`
3. 统一使用专家 #1 进行所有测量
4. 重新运行 microbench 获取准确数据

## 代码修改位置
- `microbench.py:228-236` — GPU 测量增加预热和多次测量
- `microbench.py:178` — 拷贝测量改为 `range(1, 2)` 与 GPU/CPU 测量一致
- `deepseek.py:1988` — 删除 `torch.cuda.synchronize()`
- 其他模型的 `run_expert_at_cpu` 同理检查

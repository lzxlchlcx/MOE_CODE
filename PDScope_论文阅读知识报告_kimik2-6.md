# PDScope 论文阅读知识报告

## 一、论文基本信息

- **标题**: PDScope: PD-Perceptive Single-Batch MoE Inference for Resource-Constrained Hardware
- **关键词**: Mixture-of-Experts (MoE), Offloading, LLM Inference, Phase Heterogeneity
- **核心贡献**: 首个面向阶段的感知推理系统，通过全栈协同设计解决资源受限硬件上单批次 MoE 推理的内存墙和 PCIe 延迟墙问题

---

## 二、研究背景与核心问题

### 2.1 MoE 模型推理的困境

混合专家（MoE）模型通过稀疏激活机制实现了大语言模型的高效扩展，但在资源受限硬件（如单卡 GPU + CPU）上部署时面临两大根本瓶颈：

1. **内存墙**: 专家参数量巨大，单卡显存放不下全部专家，必须借助 CPU 内存进行 offloading
2. **PCIe 延迟墙**: 专家权重在 CPU 内存和 GPU 显存之间传输时，PCIe 带宽成为瓶颈

### 2.2 阶段异构性（Phase Heterogeneity）

论文指出，现有系统的根本缺陷在于**忽视了一个关键事实**：LLM 推理的两个阶段——Prefill（预填充）和 Decode（解码）——在计算特性上存在根本差异：

| 维度 | Prefill 阶段 | Decode 阶段 |
|---|---|---|
| 计算特性 | 计算密集型（大量 Token 并行处理） | 访存密集型（逐 Token 自回归生成） |
| 专家激活 | 高度不平衡（不同专家处理 Token 数差异大） | 绝对均衡（每个激活专家只处理 1 个 Token） |
| 优化目标 | 最大化 GPU 利用率，隐藏 I/O 延迟 | 最小化每 Token 延迟，平衡 CPU/GPU 负载 |
| 瓶颈所在 | I/O 与计算的重叠度不足 | 微观系统开销（内核启动、缓存未命中）占主导 |

这种**阶段异构性**导致"一刀切"的优化策略在两个阶段表现截然相反。例如：
- 静态放置策略在 Decode 阶段表现尚可，但在 Prefill 阶段导致 GPU 空闲
- 按需加载策略在 Prefill 阶段能重叠 I/O 与计算，但在 Decode 阶段产生严重的 I/O 气泡

## 三、三大核心创新详解

### 3.1 PhasePer：阶段感知可学习预测器

#### 设计动机
传统启发式预测方法（基于统计、门控网络、Token 相似度）存在一个致命假设：专家激活信号与层/阶段无关。但论文通过实验发现：
1. **跨阶段偏差**: 同一 Token 在 Prefill 和 Decode 阶段的专家激活模式存在系统性显著差异
2. **层组效应**: 输入层/输出层与中间层的激活统计特征截然不同（路由权重偏斜度、层间隐藏状态余弦相似度）

#### 架构设计
PhasePer **不是单个预测器，而是一个预测器系统**：

```
对于 N 层 MoE 模型，PhasePer 维护：
- N 个 Prefill 专用预测器（每层一个）
- N 个 Decode 专用预测器（每层一个）
```

**分层自适应架构**:
- **输入/输出层组**: 激活高度倾斜且噪声大 → 使用浅层网络防止过拟合
- **中间层组**: 层间特征稳定 → 使用更深网络 + 门控残差块，由隐藏状态动态调节

#### 输入输出
- **输入**: 当前层完整状态三元组 $(a, w_i, e_i)$
  - $a$: Attention 输出的隐藏状态
  - $w_i$: 门控网络计算的路由权重（连续值，0~1）
  - $e_i$: 实际激活的专家 ID（离散索引）
- **输出**: 下一层每个候选专家被激活的预测概率分布 $p_i$

> **$w_i$ 与 $e_i$ 的区别**:
> - $e_i$ 决定"选哪个专家干活"（Top-k 离散选择）
> - $w_i$ 决定"专家成果占多大比重"（加权求和时的系数）

#### 训练策略
- **数据隔离**: 在 ShareGPT 数据集上离线运行 128 batch 收集数据，严格按 Prefill/Decode 分离为两个训练集
- **损失函数定制**:
  - Prefill 输入/输出组: 混合损失 $\mathcal{L}_{prefill}=\mathcal{L}_{expert}+\lambda\mathcal{L}_{focal}$（对抗类别不平衡）
  - Decode 输入/输出组: 仅 Focal 损失 $\mathcal{L}_{decode}=\mathcal{L}_{focal}$
  - 所有中间组: 标准 BCE 损失（稳定梯度、快速收敛）
- **在线微调**: 16 个 Token 滑动窗口监控误差，超过 15% 阈值自动触发微调，耗时仅 2.1ms

#### 性能指标
| 模型 | ShareGPT Top-6 | DPO Top-6 | LLaVA Top-6 | 内存占用 | 推理延迟 |
|---|---|---|---|---|---|
| DeepSeek | 96% | 92% | 90% | 1.6-4.0MB | 0.48ms |
| Mixtral | 100% | 99% | 99% | 1.2-3.8MB | 0.15ms |
| Qwen | 97% | 94% | 92% | 1.7-4.2MB | 0.12ms |
| Moonlight | 99% | 95% | 92% | 1.6-4.0MB | 0.48ms |

### 3.2 AdaptSched：跨层自适应调度器

#### 核心思想
AdaptSched 打破了传统"逐层调度"的短视局限，通过 PhasePer 的预测获得**跨层全局视野**，在当前层放置与下一层预取之间进行协同优化。

#### Prefill 阶段：三步调度法

**Step 1: 全局排位（跨层成本建模）**

将当前层 $L$ 和预测的下层 $L+1$ 的专家放入同一个池子 $E_{all}$，按 Token 数量升序排列。对每个候选边界 $i$，计算两笔全局账：

$$
T_{all}^G = \alpha + (n + n' - i + 1) \cdot t_{io} + t_g \quad \text{(GPU 总成本: 启动延迟 + 传输 + 计算)}
$$

$$
T_{all}^C = t_c(E_{all}[0:i]) + t_{attn} \quad \text{(CPU 总成本: CPU 计算 + Attention)}
$$

当 $T_{all}^G < T_{all}^C$ 时，边界 $i$ 右侧的专家入选全局队列 $L_{global}$。

> **为什么要混合两层排序？**
> 传统逐层调度会陷入"局部最优、全局最差"：当前层可能贪婪占用全部带宽，导致下层预取不足（Underfetching）；或在 Decode 阶段盲目预取本应留在 CPU 的专家（Overfetching）。混合排序把两层需求放在同一优先级天平上权衡。

**Step 2: 局部重排（按需加载底线）**

全局队列可能包含太多 $L+1$ 层专家，导致当前层 $L$ 被"饿死"。因此在 $L_{global}$ 中**仅筛选当前层专家**，寻找最小边界 $i'$ 使得局部 GPU 成本小于局部 CPU 成本：

$$
T_G = \max[n_g \cdot t_g, \alpha + (n - i' + 1) \cdot t_{io}] + t_g
$$

$$
T_C = t_c(E_{cur}[0:i'-1])
$$

边界右侧的当前层专家构成**按需加载集合 $L_{on}$**，获得最高优先级 VIP 传输通道。

> **极端情况处理**: 即使全局队列前几位全是 $L+1$ 层大专家，Step 2 会强制"屏蔽"它们，优先确保当前层负载均衡。

**Step 3: 置信度感知预取（风险投资）**

$L_{on}$ 确定后，当前层剩余专家留给 CPU。两者的完工时间差构成 **I/O 气泡 $T_{gap} = T_C - T_G$**。

计算气泡可容纳的预取数量：

$$
f = \lfloor (T_{gap} + t_{attn}) / t_{io} \rfloor
$$

然后计算第 $|f|$ 个预取的期望效用：

$$
\xi = R_{hit} \cdot (f - |f| + 1) \cdot t_{io} - (1 - R_{hit}) \cdot (|f| - f) \cdot t_{io}
$$

- **期望收益**: 预测命中时节省的时间 × 命中率
- **期望损失**: 预测失误浪费的带宽 × 失误率
- **决策**: 仅当净收益 $\xi > 0$ 时才执行预取

#### Decode 阶段：显式负载均衡

由于单批次 Decode 每个专家只处理 1 个 Token，不存在"大小专家"之分，调度逻辑变为纯负载均衡：

$$
n_g^\rho = \arg \min \max(n_g^\rho \cdot t_g, (k - n_g^\rho) \cdot t_c)
$$

目标是使 GPU 和 CPU 同时完工。根据当前层和下一层的 GPU 驻留专家数量与 $n_g^\rho$ 的关系，触发三种模式：

| 模式 | 条件 | 动作 |
|---|---|---|
| A (保当前层) | 当前层 < $n_g^\rho$，下一层 >= $n_g^\rho$ | 暂停预取，全力按需加载当前层 |
| B (保下一层预取) | 当前层 > $n_g^\rho$，下一层 < $n_g^\rho$ | 卸载当前层多余专家，全力预取下层 |
| C (双超标清理) | 两层都 > $n_g^\rho$ | 只卸载，释放带宽 |

若两层都低于配额，则无缝降级回 Prefill 的三步逻辑。

### 3.3 系统级协同优化引擎

即使有了最优调度决策，如果执行时 CPU/GPU 握手、内存分配、内核启动耗费大量时间，收益会被系统开销吃掉。PDScope 从三个层面消除开销：

#### (1) 静态 MoE 计算图编排

**核心理念**: 用"确定性"换"速度"。把调度决策提前编译成静态执行图（DAG），运行时零分支、零动态决策。

```python
# 动态模式（传统）: 运行时做判断，CPU 阻塞
attn_out = compute_attention()
if expert_id in schedule['pcie']:
    cudaMemcpyAsync(...)
cudaStreamSynchronize()  # CPU 停下来等!
launch_moe_kernel(...)

# 静态图模式（PDScope）: 一键发射，CPU 解放
self.static_graph.launch()  # 所有操作、依赖、内存地址已绑定
```

**异步构建**: 在第 $L-1$ 层计算时，后台线程已异步建好第 $L$ 层的图。当第 $L-1$ 层算完，图纸已就绪。

**图的形状不变之谜**: 即使每层去 GPU 的专家数量变化，CUDA Graph 的拓扑也不变——因为 GPU 侧永远只有一个 **Batched MoE Kernel** 节点。变化的只是输入的"元数据数组"（专家地址列表、Token 偏移、激活数量），通过指针替换即可适配。

#### (2) 图引导的缓存预取

- **CPU 端**: 在 Attention 计算期间，后台将预测专家的权重从主存预取到末级缓存（L3 Cache）。Attention 算完时，数据已在缓存中"热身"完毕
- **GPU 端**: 将所有驻留专家的任务打包，在 Attention 和 MoE 间隙**一次性批量提交内核**，消除多次内核启动开销

#### (3) 零拷贝异步 I/O 优化

- **显存预注册**: 启动时一次性在 GPU 中注册缓冲区（按需加载用双缓冲，预取用乒乓缓冲），运行时不再调用昂贵的 `cudaMalloc`
- **三流并发**: 每个专家由 3 个矩阵组成（gate_proj, up_proj, down_proj），使用 3 个独立 CUDA Stream 同时传输，填满 PCIe 通道
  - 单流的问题: 即使矩阵很大，Copy Engine 的"在途未确认请求数"有限，且存在微观断流（Micro-stalls）
  - 多流的优势: 一个流"喘息"时，硬件调度器无缝切换到另一个流，实现"见缝插针"
- **零拷贝 + 异步**: 全程使用锁页内存（Pinned Memory），DMA 硬件直接从内存"吸"数据到 GPU，CPU 完全不参与复制，下达指令后即可转身干别的

**PCIe 带宽利用率**: 66%–80%，接近物理极限。

#### 后台调度线程的物理隔离

```python
# 主线程绑定到计算核心 0-15
os.sched_setaffinity(0, {0,1,2,...,15})

# 后台调度线程绑定到独立核心 31
# 通过 Linux 的 sched_setaffinity / pthread_setaffinity_np 实现物理隔离
```

两个线程通过无锁环形缓冲区（Lock-free Ring Buffer）通信，主线程发状态、后台线程返图纸。

| 组件 | Prefill 增益 | Decode 增益 | 说明 |

1. 即使非完美预测，只要对"热专家"的命中率高于随机基线（如 Mixtral 的 25%），预取就有统计收益
2. AdaptSched 的局部重评估机制在预测错误时回退到按需加载，防止级联错误

## 五、关键技术问答（对话深度讨论精华）

### Q1: PhasePer 的两个预测器如何同时使用不同损失函数且都能收敛？

**答**: PhasePer 在物理上**不是"同一个"预测器**，而是为 Prefill 和 Decode 分别实例化了**完全独立的模型**。训练数据严格隔离，各自使用专属损失函数更新各自的权重。推理时根据当前阶段动态"切换"调用对应的预测器。

### Q2: AdaptSched 的调度会产生阻塞或性能影响吗？

**答**: **完全不会**。调度计算在独立的**后台线程**中异步执行，且被绑定到独立的 CPU 物理核心。当 GPU/主线程计算第 $L$ 层时，后台线程已在计算第 $L+1$ 层的调度方案。调度算法的复杂度仅为 $O(N)$ 标量运算（几微秒~几十微秒），远小于 GPU 计算一层的时间（几毫秒）。

### Q3: 为什么需要 3 个 CUDA Stream 而不是 1 个？

**答**: 即使矩阵很大，单流也无法跑满 PCIe 带宽，原因有三：
1. **Copy Engine 并发请求数限制**: 硬件有"在途未确认请求"上限，单流可能填不满
2. **微观断流**: 单流传输块之间存在几微秒的"汇报→领新任务"间隙
3. **多物理引擎利用**: A100 等企业卡有 2-3 个物理 Copy Engine，多流可同时利用

3 个流刚好对应专家内部的 3 个矩阵（gate/up/down），是"顺应数据结构 + 硬件饱和"的黄金甜点。

### Q4: CUDA Graph 的拓扑结构固定，如何适应每层变化的专家数量？

**答**: 现代深度学习系统使用 **Batched/Fused Kernel**。GPU 侧永远只有一个巨大的 Kernel 节点，输入是"元数据数组"的指针（专家地址列表、Token 偏移、数量）。数量变化时，只需更新数组内容（指针替换），Kernel 内部循环处理。这被称为 **"宏观静态拓扑 + 微观数据驱动"**，将动态控制流降维为静态数据流。

### Q5: 如果 PhasePer 预测持续出错，系统会崩溃吗？

**答**: 不会。系统有多重安全网：
1. **期望效用过滤**: 当预测置信度低时，$\xi < 0$，调度器自动放弃预取，回退到按需加载
2. **局部重评估**: 即使预取了错误专家，当前层的边界计算基于实际 Token 数，不受影响
3. **最差情况**: 性能退化为 HybriMoE 级别，不会低于基线

## 六、核心概念速查表

| 术语 | 含义 |
|---|---|
| MoE (Mixture of Experts) | 混合专家模型，每层通过门控网络选择 Top-k 专家参与计算 |
| Prefill | 预填充阶段，处理输入 prompt，计算密集型 |
| Decode | 解码阶段，逐 Token 生成输出，访存密集型 |
| Phase Heterogeneity | 阶段异构性，Prefill 和 Decode 在计算/I/O 特性上的根本差异 |
| Offloading | 将模型参数从 GPU 显存卸载到 CPU 内存，以突破显存限制 |
| Top-k | 门控网络选择的排名前 k 的专家 |
| Hot/Cold Experts | 热/冷专家，指被分配处理 Token 数量多/少的专家 |
| I/O Bubble | I/O 气泡，计算等待数据传输产生的空闲时间 |
| Pinned Memory | 锁页内存，防止 OS 换页，允许 DMA 直接传输 |
| CUDA Stream | CUDA 流，独立的 GPU 任务队列，支持并发执行 |
| CUDA Graph | CUDA 图，预捕获的 GPU 操作序列，消除运行时开销 |
| Batched Kernel | 批处理内核，将多个小任务合并为一个大 Kernel 启动 |
| Copy Engine | GPU 上的 DMA 拷贝引擎，负责 Host-Device 数据传输 |

## 七、总结与启发

PDScope 的核心方法论是 **"阶段感知 + 全栈协同"**：

1. **上层**: PhasePer 利用阶段异构性的可预测性，将专家激活从"不确定性"转化为"可调度资源"
2. **中层**: AdaptSched 以跨层全局视角协同优化放置与预取，打破逐层调度的局部最优陷阱
3. **下层**: 系统级优化（静态图、缓存预取、零拷贝 I/O）将智能决策转化为零开销执行

**工程启示**:
- **延迟隐藏（Latency Hiding）**: 用廉价的 CPU 后台微秒级算力，换取昂贵的 GPU/PCIe 毫秒级流畅
- **确定性换速度**: 用静态图消除运行时动态决策开销
- **软硬件协同**: 软件设计必须迎合硬件物理特性（如 PCIe 通道数、Copy Engine 数量）
- **数据流降维**: 将动态控制流转化为静态数据流，是现代推理框架的核心设计模式
